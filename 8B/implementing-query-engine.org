* Overview: The Engine Under the Logic

In the previous lecture, we saw *what* the query language does
(declarative logic programming). Now, we examine *how* it is actually
implemented.

At the very bottom, the engine relies on two core mechanisms:

1. A *Pattern Matcher* (to compare queries against the database).
2. A *Stream Processing Architecture* (to handle multiple possible
   answers).

* Pattern Matching: Variables and Literals

In the logic syntax, literal data matches itself, while variables are
denoted by a question mark (e.g., ~?x~, ~?person~).

A pattern matcher is a function that takes three inputs:

1. A *Pattern* (e.g., ~(job ?x (computer . ?type))~)
2. A *Datum* from the database (e.g., ~(job (Bitdiddle Ben) (computer wizard))~)
3. A *Dictionary* (often called a *Frame*) of existing variable bindings.

If the match is successful, the matcher returns an *extended*
dictionary. If it fails, it returns a special failure signal.

* The Core Concept: Streams of Frames

How does the system handle multiple possible answers without getting
tangled? It uses a pipeline of streams.

A *Primitive Query* (like asking for someone's supervisor) acts as a
black box.

- The input to the box is a *Stream of Frames* (a sequence of
  dictionaries).
- For every frame that enters the box, the query checks the database
  to see if it can satisfy the pattern *given the current bindings in
  that frame*.
- Because one frame might match multiple database entries, a single
  incoming frame can result in *multiple* outgoing frames.
- If it doesn't match anything, zero frames come out (the frame is
  discarded).


* Implementing Compound Queries (Plumbing)

Because every query takes a stream of frames and returns a stream of
frames, they have a uniform interface. They can be plugged together
like plumbing or electronic filters.

** AND (Series / Pipeline)

To implement ~(AND Query1 Query2)~:

- The initial stream of frames is fed into ~Query1~.
- The *output* stream of ~Query1~ is directly piped as the *input* stream to
  ~Query2~.
- Any frame that survives both boxes satisfies both conditions.

** OR (Parallel / Merge)

To implement ~(OR Query1 Query2)~:

- The *same* input stream is fed into both ~Query1~ and ~Query2~
  independently.
- The system then interleaves or merges their output streams together
  into a single stream.

** NOT (Filter)

To implement ~(NOT Query)~:

- It acts as a filter on the stream. For each incoming frame, it
  attempts to satisfy the query.
- If the query produces *any* matches for that frame, the frame is
  *discarded* (because the condition was proven true, making the NOT
  false).
- If the query produces *no* matches, the original frame is passed
  through unaltered.

* Elisp Demonstration: Simulating the Pipeline

We can simulate this "Stream of Frames" concept in Emacs Lisp using
lists of association lists (alists). Evaluate this block to see how an
empty frame is expanded by a query, and then filtered by a second
query.

#+BEGIN_SRC elisp
  (setq database '(((job . "wizard") (name . "Ben"))
                   ((job . "programmer") (name . "Alyssa"))
                   ((job . "trainee") (name . "Louis"))))

  ;; Our starting point: A stream containing one empty frame
  (setq initial-stream '( () ))

  (defun query-all-employees (input-stream)
    "Simulates a query that matches every employee in the DB."
    (let ((output-stream nil))
      (dolist (frame input-stream)
        (dolist (entry database)
          ;; We extend the empty frame with the DB entry
          (push (append entry frame) output-stream)))
      output-stream))

  (defun query-not-trainee (input-stream)
    "Simulates (NOT (job ?x trainee)) acting as a filter."
    (let ((output-stream nil))
      (dolist (frame input-stream)
        ;; If the frame says they are a trainee, drop them.
        ;; Otherwise, pass them through to the output stream.
        (unless (string= (cdr (assoc 'job frame)) "trainee")
          (push frame output-stream)))
      (nreverse output-stream)))

  ;; The Pipeline: AND (query-all-employees) (query-not-trainee)
  (princ "Running Stream Pipeline:\n")
  (let* ((stream-1 (query-all-employees initial-stream))
         (stream-2 (query-not-trainee stream-1)))
    (dolist (final-frame stream-2)
      (princ (format "Result Frame: %s\n" final-frame))))
#+END_SRC

#+RESULTS:
: Running Stream Pipeline:
: Result Frame: ((job . programmer) (name . Alyssa))
: Result Frame: ((job . wizard) (name . Ben))

* The Limits of Pattern Matching

In the previous segment, we used a *Pattern Matcher* to compare a
query containing variables (e.g., ~?x~) against a literal datum in the
database.

However, pattern matching is strictly a one-way street. The database
entry cannot contain variables. To process logical *Rules* (which
define relationships abstractly and have variables of their own), we
need a more powerful, two-way matching mechanism called
*Unification*.

* Memory Management of Dictionaries (Frames)

During the Q&A that opens this segment, a student asks about the
lifecycle of the dictionaries (frames) generated by the query system.

- *Creation:* When a query processes a stream, the matcher creates new
  dictionaries dynamically to hold the new variable bindings.
- *Destruction:* These dictionaries are temporary. Once they flow
  through the pipeline and are no longer needed (e.g., a frame is
  discarded by a ~NOT~ filter, or printed to the screen), they become
  "garbage" and are automatically cleaned up by Lisp's Garbage
  Collector.

* Unification: Two-Way Matching

Unification is the generalized form of pattern matching. It takes two
patterns, *both* of which can contain variables, and attempts to find a
set of bindings that makes them identical.

** Example of Unification

Suppose we want to unify these two patterns:
1. ~(?x a ?y)~
2. ~(?y ?z a)~

The Unifier processes them element by element:

- Element 1: ~?x~ unifies with ~?y~. The system notes that ~?x~ and
  ~?y~ must be the same thing.
- Element 2: ~a~ unifies with ~?z~. The system binds ~?z~ to the
  literal ~a~.
- Element 3: ~?y~ unifies with ~a~. Since ~?x~ is tied to ~?y~, now
  ~?x~, ~?y~, and ~?z~ are all bound to ~a~.

If the system encounters a contradiction (e.g., trying to bind ~?x~ to
~b~ when it is already bound to ~a~), the unification *fails*.

** The Hardware Analogy

Think of unification like connecting wires on a breadboard.

- Pattern matching is like reading a sensor: data flows from the
  sensor into your variable.
- Unification is like tying two bi-directional bus lines together. If
  you tie line ~A~ to line ~B~, and later drive line ~B~ high, line ~A~ is
  also driven high. They are logically fused.

* Applying Rules

Rules act like the "procedures" of our logic language. They consist of
a *Conclusion* (the header) and a *Body* (the logic).

When a query asks something that requires a rule, the evaluator
follows these steps:

1. *Unify the Query with the Conclusion:* The system takes the
   incoming query frame and unifies it with the rule's conclusion.
2. *Variable Renaming:* To prevent collisions (what if the query uses
   ~?x~ and the internal rule also uses ~?x~ for a completely
   different purpose?), the system renames all variables inside the
   rule to unique, temporary names before unification.
3. *Evaluate the Body:* If unification succeeds, the resulting
   extended frame is passed into the rule's *Body*. The body is
   evaluated just like a compound query (~AND~, ~OR~, etc.), which
   eventually yields the final stream of answers.

* The Danger of Infinite Loops

Because the logic evaluator heavily relies on depth-first search to
evaluate these streams, it is vulnerable to infinite loops.

If a rule is defined recursively (e.g., "A is the boss of B if A
manages B, OR if A manages someone who is the boss of B"), the system
might continuously expand the rule infinitely without ever checking
the base facts. This is a known limitation of the simple depth-first
evaluation strategy. Professional logic systems (like Prolog) often
use more advanced search strategies to mitigate this, but it remains a
fundamental challenge in declarative programming.

* Elisp Demonstration: A Simple Unification Trace

While a full unifier is complex, we can trace the basic logic of
variable binding in Elisp to see how two variables become "fused" in
an association list.

#+BEGIN_SRC elisp
  (setq current-frame nil)

  (defun extend-if-possible (var val frame)
    "Binds VAR to VAL in the FRAME, checking for contradictions."
    (let ((existing-binding (assoc var frame)))
      (if existing-binding
          (if (equal (cdr existing-binding) val)
              frame ;; Already bound to the same value, OK
            'failed) ;; Contradiction!
        (cons (cons var val) frame)))) ;; Add new binding

  ;; Step 1: Bind ?x to 'a
  (setq current-frame (extend-if-possible '?x 'a current-frame))

  ;; Step 2: Try to bind ?x to 'b (This will fail)
  (princ (format "Attempting contradiction: %s\n"
                 (extend-if-possible '?x 'b current-frame)))

  ;; Step 3: Bind ?y to 'a (This succeeds)
  (setq current-frame (extend-if-possible '?y 'a current-frame))
  (princ (format "Final Frame: %s\n" current-frame))
#+END_SRC

* The Reality of the Logic Engine

While the mathematical theory of logic programming is elegant, turning
it into a working machine involves practical compromises. In this
segment, Professor Sussman explains how the query language manages
variable collisions and addresses the severe limitations of its
evaluation strategy.

* The Variable Collision Problem

When the system applies a rule, it evaluates the body of the rule. But
what happens if the user's query uses a variable named ~?x~, and the
internal definition of the rule *also* uses ~?x~ for a completely
different purpose?

** The "Brutal" Solution: Variable Renaming

In standard Lisp, Python, or C, this problem is solved using
environments (Stack Frames). A local variable inside a function
shadows the global variable, keeping them separate.

However, the logic query system doesn't build frames for rule
application. Instead, it uses a "brutal" but conceptually simple
string-replacement tactic. Every time a rule is applied, the system
generates a unique identifier (like an incrementing integer) and
aggressively renames every variable inside the rule to ensure it is
globally unique.

- ~?x~ becomes ~?x-1~
- ~?y~ becomes ~?y-1~

Sussman points out that Lisp *could* have been implemented this way (by
renaming variables during the substitution model instead of using
environments), but it is computationally inefficient because it
requires traversing and rewriting the entire procedure body every time
it is called.

* The lisp-value Trapdoor

A student asks a perceptive question: Does using ~lisp-value~ (the
trapdoor that allows logic queries to run standard Lisp code) ruin the
bi-directional nature of the logic language?

Sussman confirms that it absolutely does.

- A pure logic statement like ~(job ?x ?y)~ is multi-directional. You
  can ask "Who is a programmer?" or "What is Ben's job?".
- But ~(lisp-value > ?salary 30000)~ is strictly one-way. Lisp's ~>~
  operator is a procedural function that *demands* two numbers as
  input. It cannot "solve" for ~?salary~.

If the query engine hits a ~lisp-value~ filter before the variables
inside it are bound to actual values, the underlying Lisp interpreter
will crash.

* The Order of Evaluation: Logic vs. Time

In pure mathematical logic, ~A AND B~ is strictly equivalent to ~B AND
A~. However, in our logic programming system, the order is a matter of
life and death for performance.

** The Pipeline Effect

Because ~(AND Query1 Query2)~ pipes the output of Query1 into Query2:

- If Query1 is ~(job ?x computer-programmer)~ (which finds 3 people) and
  Query2 is ~(supervisor ?x ?y)~, the system only has to check the
  supervisor relation 3 times.
- If you reverse it, the system must find the supervisors of *every
  person in the company* first, and then filter that massive list down
  to just the programmers.

** Infinite Loops and Depth-First Search

The query system relies on *Depth-First Search (DFS)* to process the
streams. When it encounters an ~OR~ or a recursive rule, it fully
explores one branch before looking at the other.

If a rule is defined recursively (e.g., "A is the boss of B if A
manages B, OR if A manages someone who is the boss of B"), the DFS
evaluator might dive down the recursive branch infinitely, continually
expanding the rule without ever pausing to check the base
facts.

This means a declarative program might be perfectly logically correct,
but completely un-executable by the interpreter. Writing logic
programs requires thinking carefully about the procedural mechanism
executing them.

* Elisp Demonstration: The "Brutal" Renamer

We can easily replicate the variable renaming algorithm in Emacs
Lisp. Think of this as analogous to how the C preprocessor expands
macros, substituting symbols before actual execution begins.

#+BEGIN_SRC elisp
  (defun rename-variables (rule-body id)
    "Recursively renames logic variables by appending an ID."
    (cond ((null rule-body) nil)
          ((symbolp rule-body)
           (let ((name (symbol-name rule-body)))
             ;; If the symbol starts with "?", it's a variable
             (if (string-prefix-p "?" name)
                 (intern (format "%s-%d" name id))
               rule-body)))
          ((listp rule-body)
           ;; Recursively traverse the list structure
           (cons (rename-variables (car rule-body) id)
                 (rename-variables (cdr rule-body) id)))
          (t rule-body)))

  ;; Let's test the brutal renaming on a mock rule
  (setq mock-rule '(rule (boss ?x ?y) (manages ?y ?x)))

  (princ (format "Original Rule: %s\n" mock-rule))
  ;; Apply the rule with a unique ID of 42
  (princ (format "Renamed Rule:  %s\n" (rename-variables mock-rule 42)))
#+END_SRC

#+RESULTS:
: Original Rule: (rule (boss 120 121) (manages 121 120))
: Renamed Rule:  (rule (boss 120 121) (manages 121 120))

* Mitigating the Infinite Loop

At the end of the last segment, we saw how the query system's
Depth-First Search (DFS) makes it vulnerable to infinite loops on
recursive rules.

Professor Sussman notes that one way to solve this is to rethink the
execution architecture. Instead of diving deep into one logical branch
until it bottoms out (the standard call-stack behavior you see in C or
Python), a system could evaluate all rules *in
parallel*.

** The Parallel Evaluation Strategy

If you check all rules simultaneously:
- Valid answers will "bubble up" to the surface as they are
  found.
- If one branch falls into an infinite deductive chain, you simply let
  it "buzz" endlessly in the background. Because memory and processors
  are cheap, the infinite branch won't freeze the entire program; the
  other parallel branches will still deliver the correct
  answers.

While this fixes the infinite loop freezing problem, Sussman
introduces an even deeper, more fundamental flaw in logic
programming.

* The "NOT" Problem: Negation as Failure

In pure mathematical logic, if we say ~(NOT P)~, it means we have
rigorously proven that statement $P$ is entirely false.

Our query language, however, does not work this way. When we use the
~(NOT ...)~ filter on a stream of frames, the system simply attempts
to find a match in the database. If it *fails* to find a match, it
assumes the statement is false, and therefore the ~NOT~ condition is
true.

This is known in computer science as **Negation as Failure**.

** Why is this dangerous?

If you ask the database: ~(NOT (job (Alyssa P. Hacker) (Brain Surgeon)))~
- The system looks for Alyssa being a brain surgeon.
- It doesn't find it.
- It returns *True* (She is not a brain surgeon).

But what if you ask: ~(NOT (job (Fabricio) (Software Engineer)))~
- If you haven't explicitly added your job to this specific database
  yet, the system looks for it, fails to find it, and returns *True*.
- The system confidently declares you are *not* a Software Engineer,
  simply out of its own ignorance.

* The Closed World Assumption

This flaw exists because the system relies on the *Closed World
Assumption*.

By equating "I cannot prove it is true" with "It must be false," the
machine is assuming that its internal database represents the entire,
complete universe of facts.

When you write down rules and ask what you can deduce from them, you
have to be very careful to remember that `NOT` means something
entirely different than it does in human logic. It is
an assumption that is, in reality, almost never completely
true.

** Can we fix it?

A student asks if it is possible to fix this without destroying the
ability to use ~NOT~ entirely. Sussman clarifies that there *are* ways to
implement logic systems with a "real NOT," but it requires
fundamentally altering the deduction engine so that it can handle
explicit assertions of falsehood (e.g., storing facts that state "X is
definitively NOT Y").

* Elisp Demonstration: Negation as Failure

To make this abstract concept concrete, look at how a standard lookup
function inherently relies on the Closed World Assumption. When it
returns ~nil~, we interpret that as "False", but it really just means
"Not Found".

#+BEGIN_SRC elisp
  (setq company-database '((Ben . wizard)
                           (Alyssa . programmer)))

  (defun is-programmer? (person)
    "Checks if a person is a programmer."
    (eq (cdr (assoc person company-database)) 'programmer))

  ;; Correct deduction:
  (princ (format "Is Alyssa a programmer? %s\n"
                 (if (is-programmer? 'Alyssa) "Yes" "No")))

  ;; The Closed World trap:
  ;; Because 'Fabricio' is not in the database, the assoc function
  ;; returns nil. We treat this 'nil' as "No", even if in the
  ;; real world, it might be true!
  (princ (format "Is Fabricio a programmer? %s\n"
                 (if (is-programmer? 'Fabricio) "Yes" "No")))
#+END_SRC
